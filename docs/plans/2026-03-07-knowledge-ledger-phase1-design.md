# Knowledge Ledger Phase 1 -- Design Document

**Date**: 2026-03-07
**Status**: Approved
**Parent**: `docs/plans/2026-03-07-ai-intelligence-layer-design.md`

## Goal

Build the Knowledge Ledger foundation: a multi-entity knowledge base with Q&A on requisitions, auto-capture from existing flows, pre-computed AI insights, and expiry logic. This is Phase 1 of the 5-phase AI Intelligence Layer.

## Decisions

| Decision | Choice |
|----------|--------|
| Entity scope | Full multi-entity (MPN, vendor, company, req, requirement) |
| Q&A routing | Sales selects 1+ buyers; exchange visible to all |
| AI sidebar trigger | Pre-computed + refresh button |
| Entry types | Manual + AI + auto-capture from quotes/offers/RFQ responses |
| Linkage model | Nullable FK columns on knowledge_entry |
| Expiry | Full logic (dimming, confidence weighting, configurable TTL) |
| UI placement | AI insights collapsible card on overview tab; Q&A as new tab |
| Backend pattern | Monolithic service (single knowledge_service.py) |
| Escalation/scoring | Deferred to later phase |
| Testing | Deferred until all phases complete |

## Data Model

### `knowledge_entry` table (migration 063)

| Column | Type | Notes |
|--------|------|-------|
| `id` | SERIAL PK | |
| `entry_type` | VARCHAR(20) NOT NULL | `question`, `answer`, `fact`, `note`, `ai_insight` |
| `content` | TEXT NOT NULL | The knowledge content |
| `source` | VARCHAR(20) NOT NULL | `manual`, `ai_generated`, `system`, `email_parsed`, `teams_bot` |
| `confidence` | FLOAT NULL | For AI-generated entries (0.0-1.0) |
| `expires_at` | TIMESTAMP NULL | Price facts auto-set (e.g., +90 days) |
| `is_resolved` | BOOL DEFAULT false | For Q&A -- marks question as answered |
| `parent_id` | INT FK(self) NULL | Answer -> Question threading |
| `assigned_to_ids` | JSON DEFAULT '[]' | User IDs selected by sales for Q&A routing |
| `created_by` | INT FK(users) NOT NULL | Who created it |
| `mpn` | VARCHAR NULL | Part number linkage |
| `vendor_card_id` | INT FK NULL | Vendor linkage |
| `company_id` | INT FK NULL | Customer linkage |
| `requisition_id` | INT FK NULL | Req linkage |
| `requirement_id` | INT FK NULL | Line-item linkage |
| `created_at` | TIMESTAMP DEFAULT now() | |
| `updated_at` | TIMESTAMP DEFAULT now() | |

### Indexes

- `ix_ke_requisition` on `(requisition_id, created_at)` -- main query path
- `ix_ke_mpn` on `(mpn)` -- cross-req MPN lookup
- `ix_ke_company` on `(company_id, created_at)` -- company-level knowledge
- `ix_ke_vendor` on `(vendor_card_id)` -- vendor intelligence
- `ix_ke_parent` on `(parent_id)` -- Q&A thread loading
- `ix_ke_expires` on `(expires_at)` WHERE `expires_at IS NOT NULL` -- expiry cleanup

### Expiry defaults

- Price facts from quotes/offers: 90 days
- Lead time facts: 180 days
- Notes/Q&A: no expiry
- AI insights: 30 days (re-generated by background job)

## Knowledge Service (`app/services/knowledge_service.py`)

### CRUD

- `create_entry(db, user_id, entry_type, content, source, links, ...)` -> KnowledgeEntry
- `get_entries(db, filters)` -> list (filters by any FK, entry_type, expired/active)
- `get_thread(db, question_id)` -> question + answers
- `update_entry(db, entry_id, user_id, ...)` -> KnowledgeEntry
- `delete_entry(db, entry_id, user_id)`

### Q&A

- `post_question(db, user_id, content, assigned_to_ids, links)` -> creates question + notifications
- `post_answer(db, user_id, question_id, content)` -> creates answer, marks resolved, notifies asker

### Auto-capture

- `capture_quote_fact(db, quote)` -> price/vendor/MPN fact, expires_at=+90d
- `capture_offer_fact(db, offer)` -> from parsed RFQ responses
- `capture_rfq_response(db, response)` -> lead time, MOQ, availability facts

Called directly from existing services (quote creation, offer parsing, RFQ response handling).

### AI Context Engine

- `build_context(db, entity_links)` -> gathers all knowledge for given entities, formats for prompt
- `generate_insights(db, requisition_id)` -> claude_structured() with Sonnet, thinking 5000, returns 3-5 insights
- `get_cached_insights(db, requisition_id)` -> returns pre-computed insights

Context gathering pulls 5 layers:
1. Direct knowledge linked to this req
2. MPN knowledge from any req sharing the same MPNs
3. Vendor knowledge for vendors who've quoted on this req
4. Company knowledge for the req's customer
5. Expired entries included but marked `[OUTDATED]`, weighted 0.3x

## Background Jobs

| Job | Schedule | Purpose |
|-----|----------|---------|
| `refresh_active_insights` | Every 6h | Re-generate insights for reqs updated in last 24h, cap 50/run |
| `expire_stale_entries` | Daily 3AM | Mark expired entries for UI dimming + lower AI weight |
| `backfill_knowledge` | One-time | Scan existing quotes/offers to seed initial facts |

### Cost control

- 50 reqs max per insight refresh run (most recently active first)
- Skip reqs unchanged since last insight generation
- Haiku for fact extraction, Sonnet only for insight synthesis

## Frontend UI

### AI Insights Card (overview tab, collapsible)

Positioned at top of requisition overview tab:
- Shows 3-5 pre-computed insights
- Refresh button triggers re-generation with spinner
- Expired facts shown dimmed with "(may be outdated)"
- Collapsed/expanded state persisted in localStorage
- Empty state: "No insights yet. Click Refresh to generate."

### Q&A Tab (new tab in requisition drawer)

- "Ask Question" button opens modal with text area + multi-select buyer picker
- Questions show: content, author, time, assigned buyers, resolved status
- Answers nested under their parent question
- "Answer" button visible on unanswered questions
- Notes (type=note) shown in timeline without threading
- System facts (auto-captured) shown with subtle "auto" badge styling
- Filters: All / Questions / Notes / Facts
- Expired facts dimmed with "(may be outdated)" label and days-until/since expiry

### Notifications (in-app only for Phase 1)

- New question -> notification to each assigned buyer
- Answer posted -> notification to original asker
