### [WF1-workflow/buyplan-approvals] BP-1 — All five buy-plan detail modals (Submit, Approve, Reject, Halt, Cancel) are dead buttons — hx-post inside Alpine <template x-if> is never processed by htmx
- **Where:** `app/templates/htmx/partials/buy_plans/detail.html:161`
- **Evidence:** detail.html:161-335 — every modal body is a `<template x-if="modalType === '...'">` whose confirm button carries hx-post (submit :183, approve :246, reject :271, halt :300, cancel :325). htmx 2.0.10 (package.json) processes nodes only at load/after its own swaps; template content is an inert DocumentFragment, and Alpine's x-if clone is never handed to htmx.process — no hook exists (grep of app/static/htmx_app.js shows htmx.process only at :1795 and :2646, both unrelated; the htmx:afterSwap handler at :1168 only calls Alpine.initTree for 4 hardcoded ids). Empirically verified with the real dist bundle (htmx_app-B2Fhatrr.js): the cloned button fires zero htmx:beforeRequest events on click; after htmx.process() it fires. The Issue modal (detail.html:222) works only because it uses htmx.ajax() in @click, and its own comment shows the author assumed hx-post-in-x-if is live. Net effect: from the detail screen a draft can never be submitted, a pending plan can never be approved/rejected (reject has NO other UI path — My Queue's Reject is a 'Review' link back to this dead modal, _surface_my_queue.html:125), and Halt/Cancel silently close the modal and do nothing.
- **Suggested fix:** Convert each modal confirm button to the Issue-modal pattern (htmx.ajax('POST', url, {target:'#main-content', values:{...}}) in @click), or replace the x-if templates with x-show blocks so the forms exist in the server-rendered DOM at swap time, or add x-init="htmx.process($el)" on each x-if root. Then audit the other 14 templates using `<template x-if>` + hx-* for the same defect.
