# Microsoft Graph / 365 Integration — Tightness Assessment

## 1. The integration map

**Auth / client.** One canonical `GraphClient` (`app/utils/graph_client.py:36`) wraps all Graph HTTP with retry (3x, exp backoff 2/4/8s), Retry-After honoring on 429/503, immutable-ID header (H1), delta-query state machine (410→`GraphSyncStateExpired`), and a TESTING short-circuit (retries=0, but still issues the HTTP call). Two token paths feed it: delegated per-user OAuth refresh tokens (`token_manager.py`, refreshed by a 15-min scheduler job with Redis locks) and an app-only client-credentials token for the SharePoint library (`graph_app_auth.py`). `GRAPH_SCOPES` is a single constant. The looseness is at the edges: ~10 production callers import `get_valid_token` from the `app.scheduler` re-export shim instead of the canonical module; `attachment_service.py` / `datasheet_library.py` make raw `http.put`/`http.delete` calls bypassing GraphClient entirely; several callers (`_sync_user_contacts`, `crm/offers.py` OneDrive, `sources.py`) use `user.access_token` directly without `get_valid_token`; and two import-level dead references exist (`get_graph_client()` in `prospect_scheduler.py:172`, `list_messages()` in `prospect_discovery_email.py:102`) that fire only inside swallowed `except` blocks.

**Send.** 12 distinct outbound send sites, all `POST /me/sendMail`. Primary vendor flows (`send_batch_rfq` in `email_service.py:243` and its delegators in `sightings.py`) are the solid path: hard DNC gate, per-vendor SAVEPOINT, post-send Sent-Items lookup to capture graph ids, immediate ActivityLog for the outbound clock. Beyond that the surface fragments: quote send exists as **two divergent routes** (HTMX marks-sent-without-emailing vs JSON actually-emails); follow-up, thread-reply, excess-bid, proactive, and quote sends each individually drop one or more of {DNC check, conversation-id capture, ActivityLog write}; and several routes mark success under TESTING without ever calling Graph (ghost-sent state).

**Threading / conversation.** `conversationId` + Graph message `id` are the anchors. Only the RFQ flow is fully instrumented: `Contact.graph_message_id`/`graph_conversation_id` populated via post-send lookup (`_find_sent_message`, `$top=50`) with `scan_sent_folder` as a 48h reconcile fallback. Everything else has gaps — Quote model has **no graph-id columns at all**; `ProactiveOffer.graph_message_id` exists but is never written outside the seed script; `BidSolicitation` has message-id only (no conversation-id) captured via a subject-only, `$top=5` lookup. No outbound path uses Graph `createReply`/`createReplyAll` — every "reply" is a fresh `/me/sendMail` with a manual `Re:` prefix, so threading survives only by Exchange's subject grouping.

**Inbound.** Two entry points (30-min poll `_job_inbox_scan`; 5-min webhook maintenance + real-time `handle_notification`) converge on `poll_inbox` (`email_service.py:578`). 4-tier reply matching: conversationId (global, vendor-scoped after the fact) → `[ref:N]`/`[AVAIL-N]` subject token (existence-filtered, cross-user) → exact email (user-scoped) → domain (user-scoped). Matches become `VendorResponse` rows, queue AI parse (Anthropic Batch with sequential fallback), and drive Contact status. Solid here: belt-and-suspenders dedup (`VendorResponse.message_id` unique + `ProcessedMessage`), delta queries with watermarks, SAVEPOINT-per-message, nightly cadence self-heal (`materialize_all_clocks` 04:00). Broader 365 (read-only/secondary): Drive/SharePoint attachments (two-tier user-OneDrive vs company-library), Teams (webhook logging, outbound DM, channel webhook), calendar scan, mailboxSettings, Outlook contacts delta, Teams call records, presence (built but unwired).

## 2. Where it's LOOSE — ranked

**S1 — Quote send is split across two routes with divergent behavior (CRITICAL, data corruption).** `htmx_views.py:10041` (`send_quote_htmx`) marks `quote.status=SENT` and advances `req.status=QUOTED` **without sending any email**; `crm/quotes.py:373` (`send_quote`) is the only path that actually emails. The HTMX stack is what the frontend calls — so in the live UI a "Send Quote" click can silently mark a quote sent that the customer never received. This is the single worst defect: silent, customer-facing, and corrupts requisition state.

**S2 — Quote sends have zero threading/tracking instrumentation (HIGH).** Even the real send (`crm/quotes.py:404`) carries no `[ref:]` tag, captures no graph id (Quote model has no columns, `models/quotes.py:24-84`), writes no `log_email_activity`, creates no Contact. Customer replies land as unmatched `VendorResponse status='new'` or get mis-attributed to a vendor contact via Tier-3/4. Consequence: won/lost is 100% manual, `last_reply_at` never moves on customer replies, and the whole customer-reply loop is invisible. This is the structural gap behind the quote-preflight work — the preflight has nothing to thread against.

**S3 — DNC is enforced inconsistently across send paths (HIGH, compliance).** Hard block exists in `send_batch_rfq:193` and `send_reply_htmx:3216`. It is **absent** from `send_follow_up_htmx` (`htmx_views.py:2965`), `send_email_reply` (`htmx_views.py:7508`), and `send_bid_solicitation` (`excess_service.py:938/994`). A do-not-contact vendor can be emailed through any of these three. (Quote/proactive/buyplan/ownership paths email customers or internal users, so vendor-DNC is correctly N/A there.)

**S4 — Tier-1 reply matching silently degrades on large batches (HIGH).** `_find_sent_message` (`email_service.py:466`) uses a hard `$top=50` Sent-Items window with no pagination. Any vendor beyond position 50 in a single tick gets `graph_conversation_id=NULL`, dropping all their future replies to Tier-2/3/4. Tier-3/4 are last-writer-wins single-contact, so older requisitions for a repeat vendor get no status progression. Logged as a WARNING only — no user signal. `scan_sent_folder` is the partial backstop but inherits the same window.

**S5 — Replies are new composes, not true Graph thread-replies (MEDIUM).** `send_reply_htmx` (`htmx_views.py:3246`) and `send_email_reply` (`htmx_views.py:7534`) call `/me/sendMail` with a manual `Re:` subject instead of `createReply` on the original message id; `send_email_reply` receives `conversation_id` as a form field and only echoes it to the template. Threading is coincidental (Exchange subject grouping); breaks if the vendor edits the subject. `send_email_reply` also writes no Contact and no ActivityLog, so thread-viewer replies are invisible to the timeline and unmatchable on the way back.

**S6 — TESTING bypass produces ghost-sent state (MEDIUM, test integrity).** `send_follow_up_htmx`, `send_reply_htmx` (`htmx_views.py:3021/3234`), and `rfq_send` test-branch mark contacts/responses as sent/reviewed **without any Graph call**. Tests assert on side-effects of a path that never runs in production, and the bypass is an env flag that would fire in staging if `TESTING` leaked there.

**S7 — ProactiveOffer threading is half-wired (MEDIUM).** `ProactiveOffer.graph_message_id` exists (`intelligence.py:343`) but `send_proactive_offer` (`proactive_service.py:347`) never writes it; there is no `graph_conversation_id` column at all. Proactive replies cannot be Tier-1 matched. The column is written only by `seed_proactive_demo.py`.

**S8 — Excess-bid lookup is the un-fixed twin of the RFQ lookup (MEDIUM).** `excess_service._find_sent_message` (`excess_service.py:853`) is subject-only, `$top=5`, **no recipient filter** — the exact bug that was fixed in `email_service._find_sent_message`. On a same-subject batch, all solicitations inherit whichever message is top of Sent. `BidSolicitation` also has no conversation-id column, so Tier-1 is structurally impossible for excess bids. Plus no DNC (see S3).

**S9 — `401` returns a dict instead of raising; callers mistranslate it (MEDIUM).** `GraphClient` returns `{"error":401,...}` (`graph_client.py:202`); callers like `crm/offers.py:826` treat any error key as "OneDrive item not found" → a token expiry surfaces to the user as a 404 and to developers as a 404 in logs. Combined with direct-`user.access_token` callers (`offers.py` browse, `_sync_user_contacts`, `sources.py` fallback), mid-session expiry produces a class of misleading failures with no token-expiry signal.

**S10 — Two dead import references, silently swallowed (MEDIUM latent).** `get_graph_client()` (`prospect_scheduler.py:172`) and `list_messages()` (`prospect_discovery_email.py:102`) reference functions/methods that don't exist; both fire inside `except Exception` blocks, so the prospect email-mining/discovery branches are silently 100% broken and untested.

**S11 — Cadence `last_reply_at` lags up to ~20h for inbound email (MEDIUM).** `email_received` is not in `_RULE_MEANINGFUL_TYPES` (`activity_service.py:32`), so `bump_clocks_from_activity` never advances `last_reply_at` in real time; only the 04:00 `materialize_all_clocks` heals it. Real-time bump is effectively dead for inbound email.

**S12 — Operational/edge looseness (LOW, batch).** `send_email_reply`/`rfq_send` fallback create tracking artifacts (Contact rows) with no send and no DNC; `scan_sent_folder` reconcile keys on `graph_message_id IS NULL` + 48h window with no subject, so a same-vendor resend can stamp the wrong row; webhook replay cache is in-process (`webhook_service.py:44`) and resets on restart (duplicate ActivityLog possible); OOO/bounce repair and the two classifiers (`_classify_response` vs `RESPONSE_PARSE_SCHEMA`) have parallel call sites that must be kept in sync; `token_expires_at` is hard-coded to now+1h ignoring `expires_in`; calendar scan and Teams DM creation have no delta/no chat-id reuse; Teams call records always NULL direction/phone; OOO mailbox status is fetched but never used to gate sends.

**Two cross-cutting themes the founder should hold onto:** (a) *the RFQ flow is the reference implementation and everything else is a partial copy of it* — DNC, conversation-id capture, ActivityLog write, and true-reply are present on RFQ and missing in varying combinations everywhere else; (b) *the conversation-id capture mechanism itself is fragile* (`$top=50`, no pagination, no surfaced failure) even on the one path that uses it.

## 3. What "tight" would look like

- **One send service, not 12 call sites.** A single `send_graph_email(...)` chokepoint that every flow (RFQ, follow-up, reply, quote, proactive, excess, internal) calls. It uniformly: enforces DNC for external-vendor sends (parameterized by recipient type), captures the conversation-id, writes the ActivityLog/Contact tracking row, and tags the subject. No route reimplements `/me/sendMail`.
- **One quote-send path.** Delete the silent `send_quote_htmx` mark-sent route; the HTMX button calls the real send (or the preflight then the real send). A quote is never "sent" in the DB unless an email actually left.
- **Conversation-id on every outbound, captured reliably.** Either replace the post-send Sent-Items poll with the `sendMail`-then-read-`id` pattern that paginates/uses the message id from the create response, or back it with a guaranteed `scan_sent_folder` reconcile that has no `$top` ceiling. Quote, proactive, and excess-bid models all get `graph_conversation_id` columns and write them. No silent NULL-and-degrade.
- **True threading.** Replies use Graph `createReply`/`createReplyAll` on the stored message id, not a fresh compose with a manual `Re:`. Threading survives subject edits and is matchable by conversationId on return.
- **Unified client discipline.** Every Graph HTTP goes through `GraphClient` (including attachment upload/delete) so retries/backoff/immutable-id are universal; `401` raises a typed `GraphAuthError` callers can distinguish from 404; every caller uses `get_valid_token` (no direct `user.access_token`); the `app.scheduler` re-export shim is removed and imports point at the canonical module.
- **Customer-reply loop closed.** Quote sends are tracked well enough that a customer reply matches back to the quote/requisition, advances cadence in real time, and can drive (at least suggested) won/lost.
- **Honest test mode.** TESTING mocks the HTTP layer; no route marks success while skipping the send. No ghost-sent.

## 4. Prioritized hardening options

**The single highest-leverage first move: WS-A (unify the send path).** It is the root cause behind S1, S3, S5, S6, S7, S8, and the prerequisite for S2/customer-reply work. Everything else either depends on it or is cheap cleanup after it.

**WS-A — Consolidate to one send service + kill the duplicate quote route.** *Fixes:* S1 (delete `send_quote_htmx` mark-sent; wire button to the real send), S3 (DNC enforced once, parameterized), S5 (createReply for replies), S6 (single TESTING mock point, no ghost-sent), and structurally enables S2/S7/S8 (uniform conversation-id capture + tracking write). *Effort:* L (it touches all 12 sites, but most become thin callers). *Unblocks:* every other workstream. *Dependency:* none — do this first. *Recommendation:* land S1's delete-the-route fix as a same-day hotfix even before the full refactor, given it is active silent data corruption.

**WS-B — Robust conversation-id capture.** *Fixes:* S4 (paginate/eliminate `$top=50` ceiling, or capture id from the create response), S8's lookup half, and the silent-NULL degrade. Add a user-visible signal when Tier-1 anchor is missing. *Effort:* M. *Unblocks:* reliable Tier-1 matching at scale; precondition for trusting any reply-driven automation. *Dependency:* cleaner if WS-A lands first (one capture site), but can proceed in parallel on the RFQ path.

**WS-C — Close the customer-reply loop (quotes).** *Fixes:* S2. Add `graph_message_id`/`graph_conversation_id` to Quote, tag the subject, write ActivityLog, capture the id (via WS-A/WS-B), and teach `poll_inbox` a customer/quote tier. Enables suggested won/lost and real-time customer cadence. *Effort:* M–L (model migration + matching changes). *Unblocks:* the quote-preflight effort's actual value (a tracked, reply-able quote), automated won/lost. *Dependency:* WS-A and WS-B (needs the capture mechanism it threads against).

**WS-D — Client discipline + auth honesty.** *Fixes:* S9 (`401`→typed exception, callers distinguish auth from 404), the direct-`user.access_token` callers, the `app.scheduler` shim removal, raw-http attachment calls onto GraphClient, and `token_expires_at` honoring `expires_in`. *Effort:* M (broad but mechanical). *Unblocks:* uniform retry/backoff on file ops; correct error diagnostics; refactor safety. *Dependency:* independent — can run parallel to A/B/C.

**WS-E — Dead-code + latent-failure sweep.** *Fixes:* S10 (the two non-existent function references — either implement or delete the branches and stop swallowing the exception), and decide presence-service / OOO-status (wire or drop the scope). *Effort:* S. *Unblocks:* removes silent-broken pipelines and a needless OAuth-consent scope. *Dependency:* none — cheap early win, good to bundle with WS-D.

**WS-F — Cadence + inbound correctness.** *Fixes:* S11 (make `email_received` advance `last_reply_at` in real time, or accept nightly-only and document it), S12 batch (webhook replay cache → Redis, classifier/repair call-site consolidation, scan_sent_folder reconcile keying, excess-bid conversation-id column + DNC). *Effort:* M. *Unblocks:* real-time CRM cadence accuracy; fewer duplicate activity rows. *Dependency:* the excess-bid pieces fold naturally into WS-A/WS-B if sequenced after them.

**Suggested sequence:** S1 hotfix today → WS-A → (WS-B ∥ WS-D ∥ WS-E) → WS-C → WS-F. A unblocks the most and removes the most duplication; B+C together close the customer-reply gap that the quote-preflight work actually needs; D/E/F are largely independent hardening that can fill parallel capacity.
