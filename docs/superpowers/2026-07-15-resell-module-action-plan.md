# Resell Module — Prioritized Action Plan (2026-07-15)

Synthesized from a five-lens deep review (function, workflow, data, silent-failures, performance) plus a tests lens, adversarially verified against current main (post #731/#734/#735/#736) and the live staging DB. 33 findings confirmed, 0 refuted, 34 unverified P2 polish notes. Findings that multiple lenses hit independently are merged below with the agreeing lenses named — cross-lens agreement is the strongest confidence signal in this review.

## 1. Executive summary

The Resell happy path (create → import → publish → collect offers → award → bid-back) works, but the module is **guard-free at every lifecycle edge**: publish re-opens resolved lists, award accepts withdrawn offers, and re-assembling silently destroys accepted-bid records. The **email outreach subsystem lies about failures** (failed sends recorded as "buyer ignored us", false "Emailed" activity, rows stranded in `sending` across deploys). An **entire shipped subsystem is dead in production** — the posting-window deadline/expiry/countdown machinery has no user entry point, and its only reachable render is a misleading red "Overdue" on every properly closed list. Three **anonymization leaks** (offer counts, title-probing search, outreach subject) contradict the module's own test-codified privacy rule, and **one defect is live-broken today**: bid assembly on the only active staging list 500s from dangling `best_offer_id` pointers. No finding survived verification at P0 (three filed P0s were re-graded P1 with rationale); total remediation ≈ **19–21 engineering days (~4 weeks)**, with the worst risk retired in the first week.

## 5. Product decisions needed

Genuine product calls only — everything else above has a clear engineering answer. Each with a recommendation (see feedback: options + recommendation).

1. **D1 — Is an excess posting time-boxed?** The deadline/countdown/auto-expiry/Expired-filter machinery is fully built but has no entry point (P1 #8). **Recommend:** yes — add an optional "Offers close by" date at create/publish; the job, chip, and filter then work as designed. The alternative (excise job + EXPIRED + chip + close_at branch) is more code churn for less product.
2. **D2 — Are aggregate offer counts/coverage visible to non-owner brokers?** The code documents both positions (`_header_chips.html:32-33` says public; `_lines.html` + 403 + RS-1 test say private). **Recommend:** private — the test suite already codifies count exposure as a data leak, and for single-line lists the aggregate IS the per-line datum.
3. **D3 — May a customer-ACCEPTED bid be revised?** If yes, revisions must become immutable history (new CustomerBid row per revision); if no, block re-assembly at accepted. **Recommend:** allow with history — matches the UI's own "revise as a new revision" language and preserves the migration-183 audit intent.
4. **D4 — Posted-list corrections: implement the promised "revise as a new version" flow, or drop the promise?** Determines P1 #14's scope and the fate of the dead `version` column. **Recommend:** for v1, drop the promise — honest 409 copy + draft-scope edits + close-and-recreate; re-versioning is a full feature and shouldn't block the rework.
5. **D5 — Lifecycle vocabulary:** what does "Close" mean (bid_out is captioned "Sent to the customer" even when no bid-back exists)? Is CLOSED a real terminal stage distinct from bid_out, and what do the legacy `closed`/`active`/`bidding` rows remap to? **Recommend:** rename the bid_out caption to "Closed — bids out", drop the CLOSED status and pill, remap active→open, bidding→collecting, closed→bid_out.
6. **D6 — Do offers expire?** `valid_until` is accepted by the service but never settable, enforced, or read. **Recommend:** drop it for v1 (traders withdraw manually); revisit if buyers ask for validity windows.
